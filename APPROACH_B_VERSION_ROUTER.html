Approach B: Version Router (New Layer Above Transport)

Date: 2026-06-30  |  Status: Design exploration  |  Companion: transport-changes.html, Approach A

Core idea: A new VersionRouter abstraction sits above the entire transport+protocol stack. It receives raw HTTP requests (or intercepts stdio messages), detects the protocol version, and delegates to completely independent stacks. Each version has its own transport, protocol, and server instance. The existing legacy code is wrapped as a black box — no changes required.

0. User-Facing API: End-to-End Example

A minimal example: one server with a greet tool, one client calling it. Shows exactly what application code looks like for both protocol versions under Approach B.

0.1 Today's API (for reference — 2025-11 only)

This is the current SDK API. Shown for comparison so you can see exactly what changes.

0.2 Approach B: Server serving both protocol versions

Key difference from Approach A: In Approach B, the VersionRouter is a new abstraction that sits above the transport layer. It owns transports, servers, and stacks internally. The user interacts with the router, not with transports directly. Two API styles are possible.

Style 1: Router as the primary API

import { HTTPVersionRouter } from '@modelcontextprotocol/server';
import express from 'express';
import * as z from 'zod/v4';

// 1. Create router (NEW — replaces McpServer as entry point)
const router = new HTTPVersionRouter({
  serverInfo: { name: 'greeting-server', version: '1.0.0' },
  sessionIdGenerator: () => crypto.randomUUID(),
});

// 2. Register a tool ON THE ROUTER (same signature as McpServer.registerTool)
router.registerTool(
  'greet',
  {
    description: 'Greet someone by name',
    inputSchema: z.object({ name: z.string() }),
  },
  async ({ name }) => ({
    content: [{ type: 'text', text: `Hello, ${name}!` }],
  }),
);

// 3. Wire up HTTP (router handles everything — no transport, no connect)
const app = express();
app.all('/mcp', async (req, res) => {
  const response = await router.handleRequest(toWebRequest(req));
  writeWebResponse(res, response);
});

app.listen(3000);

Style 2: McpServer as facade (preserves existing API)

The VersionRouter is not a transport. It's a new abstraction that operates at the HTTP request level, before any transport is involved. For legacy clients, it creates per-session transport+protocol stacks. For modern clients, it delegates to a single stateless transport+protocol stack. This maximizes isolation: the two version paths share nothing except the handler registry.

4.3 Session lifecycle management

The router manages the lifecycle of legacy session stacks:

class VersionRouter {
  // Session → legacy stack mapping
  private sessionStacks = new Map<string, LegacyStackEntry>();

  // Single modern stack (stateless, shared)
  private modernStack: ModernStackEntry;

  // Shared handler registrations
  private registry: HandlerRegistry;

  async handleRequest(req: Request): Promise<Response> {
    const version = detectVersion(req);

    if (version === 'modern') {
      return this.modernStack.transport.handleRequest(req);
    }

    // Legacy: look up or create session stack
    const sessionId = req.headers.get('mcp-session-id');

    if (sessionId) {
      const stack = this.sessionStacks.get(sessionId);
      if (!stack) return jsonError(404, 'Session not found');
      return stack.transport.handleRequest(req);
    }

    // No session ID: must be initialize
    return this.handleLegacyInitialize(req);
  }

  private async handleLegacyInitialize(req: Request): Promise<Response> {
    // Create a fresh legacy stack
    const transport = new WebStandardStreamableHTTPServerTransport({
      sessionIdGenerator: this.options.sessionIdGenerator,
    });
    const server = new Server(this.registry.serverInfo, {
      capabilities: this.registry.getCapabilities(),
    });

    // Wire handler registrations from shared registry
    this.registry.installHandlers(server);

    // Connect and handle
    await server.connect(transport);
    const response = await transport.handleRequest(req);

    // Store the stack keyed by session ID
    const newSessionId = transport.sessionId;
    if (newSessionId) {
      this.sessionStacks.set(newSessionId, { transport, server, createdAt: Date.now() });
    }

    return response;
  }

  // Cleanup: called when DELETE arrives or session times out
  async destroySession(sessionId: string): Promise<void> {
    const stack = this.sessionStacks.get(sessionId);
    if (stack) {
      await stack.server.close();
      await stack.transport.close();
      this.sessionStacks.delete(sessionId);
    }
  }
}
Session management complexity: The router takes on responsibility for session→stack mapping that currently lives inside the transport. This means the router must handle: session lookup, session creation on initialize, session deletion on DELETE, and optionally session timeout/cleanup. This is new code that duplicates some logic currently inside WebStandardStreamableHTTPServerTransport.

For stdio, the router intercepts the raw message stream before creating the transport. Unlike HTTP (where each request can be independently routed), stdio commits on the first message. After that, the router is effectively pass-through.

5.1 stdio implementation sketch

class VersionRouterStdio {
  private registry: HandlerRegistry;
  private locked = false;

  async start(stdin?: Readable, stdout?: Writable): Promise<void> {
    const readBuffer = new ReadBuffer();

    // Read first message to determine version
    for await (const chunk of stdin ?? process.stdin) {
      readBuffer.append(chunk);
      const msg = readBuffer.readMessage();
      if (!msg) continue;

      // Got first message: detect version and create stack
      const version = detectVersionFromBody(msg);
      const transport = new StdioServerTransport(stdin, stdout);
      const server = version === 'legacy'
        ? this.createLegacyServer()
        : this.createModernServer();

      await server.connect(transport);

      // Forward this first message
      transport.onmessage!(msg);

      // From here, stdio operates normally
      this.locked = true;
      break;
    }
  }

  private createLegacyServer(): Server {
    const server = new Server(this.registry.serverInfo, {
      capabilities: this.registry.getCapabilities(),
    });
    this.registry.installHandlers(server);
    return server;
  }

  private createModernServer(): ModernServer {
    const server = new ModernServer(this.registry.serverInfo, {
      capabilities: this.registry.getCapabilities(),
    });
    this.registry.installHandlers(server);
    return server;
  }
}
stdio challenge in Approach B: The router must intercept raw bytes from stdin, parse just enough to detect the version, then hand off to a transport. This means the router partially reimplements the transport's message reading logic (the ReadBuffer). In Approach A, this interception happens inside the transport itself, which already has the message reading infrastructure.

6.1 Client architecture

class VersionProbingClient {
  private inner: Client | ModernClient | null = null;
  private detectedVersion: ProtocolGeneration | null = null;

  async connect(transport: Transport): Promise<void> {
    try {
      // Probe: try modern discovery
      const discoverResult = await this.sendDiscover(transport);
      this.detectedVersion = 'modern';
      this.inner = new ModernClient(this.clientInfo, {
        serverCapabilities: discoverResult.capabilities,
        serverInfo: discoverResult.serverInfo,
      });
    } catch (err) {
      if (isMethodNotFound(err) || isBadRequest(err)) {
        // Server is legacy: fall back to initialize
        this.detectedVersion = 'legacy';
        this.inner = new Client(this.clientInfo, this.options);
        await this.inner.connect(transport);
        return;
      }
      if (isUnsupportedVersionError(err)) {
        // Server is modern but different version
        this.detectedVersion = 'modern';
        const supported = err.data.supported;
        this.inner = new ModernClient(this.clientInfo, {
          protocolVersion: this.selectVersion(supported),
        });
      } else {
        throw err;
      }
    }
    await this.inner!.connect(transport);
  }

  // All methods delegate to the selected inner client
  async callTool(params: CallToolParams) { return this.inner!.callTool(params); }
  async listTools(params?: ListToolsParams) { return this.inner!.listTools(params); }
  // ...etc
}

7. Shared Handler Registry

Same as Approach A: the HandlerRegistry is extracted from McpServer and shared between both server instances. However, in Approach B the registry also needs an installHandlers() method that wires its registrations into a newly-created server instance.

7.1 HandlerRegistry interface

interface HandlerRegistry {
  // Registrations
  readonly tools: ReadonlyMap<string, RegisteredTool>;
  readonly resources: ReadonlyMap<string, RegisteredResource>;
  readonly resourceTemplates: ReadonlyMap<string, RegisteredResourceTemplate>;
  readonly prompts: ReadonlyMap<string, RegisteredPrompt>;

  // Metadata
  readonly serverInfo: Implementation;
  readonly instructions?: string;
  getCapabilities(): ServerCapabilities;

  // Registration methods
  registerTool(name: string, tool: RegisteredTool): void;
  registerResource(uri: string, resource: RegisteredResource): void;
  registerPrompt(name: string, prompt: RegisteredPrompt): void;

  // Approach B specific: install into a server instance
  installHandlers(server: Server | ModernServer): void;
}

7.2 installHandlers() in Approach B

Because the router creates server instances dynamically (per-session for legacy, once for modern), it needs to wire handler registrations into each new server. This is the installHandlers() method:

class DefaultHandlerRegistry implements HandlerRegistry {
  // ...registrations...

  installHandlers(server: Server | ModernServer): void {
    // Install tools/list handler
    server.setRequestHandler('tools/list', async () => ({
      tools: Array.from(this.tools.values()).map(t => t.definition),
    }));

    // Install tools/call handler
    server.setRequestHandler('tools/call', async (request, ctx) => {
      const tool = this.tools.get(request.params.name);
      if (!tool) throw new McpError(ErrorCode.InvalidParams, `Unknown tool: ${request.params.name}`);
      return tool.handler(request.params.arguments, ctx);
    });

    // Similarly for resources/*, prompts/*
    // ...
  }
}
Dynamic registration challenge: If a tool is registered after a legacy session is already active, the existing session's server won't see it. Options: (1) re-install handlers on registration change, (2) have server delegate to registry at call time instead of install time, (3) only allow registration before start(). Option (2) is cleanest but requires the server to hold a reference to the registry.

10. Wiring: How It All Connects

10.1 User-facing API

import { VersionRouter } from '@modelcontextprotocol/server';
import { z } from 'zod/v4';

// 1. Create the router (new entry point)
const router = new VersionRouter({
  serverInfo: { name: 'my-server', version: '1.0.0' },
  legacy: {
    sessionIdGenerator: () => randomUUID(),
  },
  modern: {},
});

// 2. Register tools on the router (same API as McpServer)
router.tool('search', { query: z.string() }, async ({ query }, ctx) => {
  return { content: [{ type: 'text', text: `Results for: ${query}` }] };
});

// 3. Framework integration (new pattern)
app.all('/mcp', async (req, res) => {
  const response = await router.handleRequest(req);
  // ... send response
});

Note: the user-facing API changes from mcpServer.connect(transport) to router.handleRequest(req). The router owns transport creation internally.

10.2 Alternative: McpServer as facade

// If we want to preserve the McpServer API:
const mcpServer = new McpServer({
  name: 'my-server',
  version: '1.0.0',
});

mcpServer.tool('search', { query: z.string() }, async ({ query }, ctx) => {
  return { content: [{ type: 'text', text: `Results for: ${query}` }] };
});

// New method on McpServer that creates a VersionRouter
const router = mcpServer.createVersionRouter({
  legacy: { sessionIdGenerator: () => randomUUID() },
});

app.all('/mcp', (req, res) => router.handleRequest(req));

12. Approach B Strengths and Weaknesses

Strengths

  • Maximum isolation: Each version is a fully self-contained stack. Zero coupling between legacy and modern code paths.
  • Truly frozen legacy: The existing transport, Protocol, Server, and McpServer code can remain completely untouched. Bug fixes go into the frozen code; no architectural changes needed.
  • Clean removal path: When 2025-11 is deprecated, delete the legacy stack and the creation branch. No entanglement.
  • Independent evolution: The modern stack can be designed from scratch with no constraints from legacy architecture decisions.

Weaknesses

  • New abstraction layer: The VersionRouter is a new concept that developers must understand. It changes the integration pattern from transport.handleRequest() to router.handleRequest().
  • Session management duplication: The router partially reimplements session→stack mapping that the legacy transport already handles internally. Two layers managing related state.
  • stdio complexity: The router must intercept raw stdin bytes to parse the first message before creating a transport. This partially reimplements transport-level message reading.
  • Per-session stack overhead: Each legacy session creates a full transport + server instance. In Approach A, the legacy transport handles all sessions internally with shared infrastructure.
  • Handler installation: The installHandlers() pattern requires re-wiring handler registrations into each new server instance. Dynamic registration requires additional coordination.

13. Summary

Approach B places version detection in a new layer above the transport, treating each protocol version as a completely independent stack.

  • The VersionRouter is the single HTTP entry point. It inspects headers, manages session→stack mapping for legacy clients, and delegates to the appropriate stack.
  • Legacy code is wrapped as a black box — literally frozen with zero changes.
  • Modern code is built from scratch with no legacy constraints.
  • Shared HandlerRegistry with installHandlers() ensures tools are registered once.
  • Client-side VersionProbingClient probes server/discover and falls back to initialize.
  • MRTR transparency via exception-based continuation.

Best for: teams that prioritize zero risk to the legacy path and want maximum isolation between protocol versions, accepting the cost of a new abstraction layer and some duplication in session/stdio management.
