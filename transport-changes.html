8. SDK shape implications (summary for design discussion)

The connection model inverts: 2025-11 is "handshake establishes state, requests are lightweight." 2026-06 is "every request is self-describing, no connection state." The SDK must support both on one server instance.

Per-request scope is the core primitive. Every dispatched request needs {protocolVersion, clientCapabilities, clientInfo, logLevel, authInfo} resolved from _meta first, legacy session-store fallback, never instance fields. The server instance is a pure handler registry.

Legacy support is config-time, not request-time. initialize, setLevel, standalone GET, Mcp-Session-Id echo, SSE push for s→c — all live in a removable compat module (SessionCompat/BackchannelCompat). Wired = dual-support; unwired = 2026-06-only and those methods return -32601.

Result-type dispatch is now polymorphic. callTool etc. can return complete | input_required | task. Client needs an MRTR retry loop (core) and a result-type hook for task (extension-installed).

Subscriptions become explicit. subscriptions/listen is the only push channel; client opts in per type. list_changed becomes opt-in pull-model (with TTL hints from SEP-2549) plus optional push for clients that open a subscription.

Extensions are first-class. The registrar (#1868 + H4/H5/H6 + augmentContext) is the substrate for tasks, roots, sampling, sessions, apps, server-cards. Core gets smaller; the extension surface is where most "features" live.

9. What the spec says about backwards compatibility

The merged spec explicitly documents dual-support — this isn't an SDK invention. Citations are docs/specification/draft/basic/lifecycle.mdx and transports.mdx at 85ec3771.

Servers supporting both client generations

"A server that wishes to support both legacy clients (which expect an initialize handshake) and modern clients (which use per-request metadata) MAY implement both behaviors."

Concretely, a dual-support server keeps:

  • An initialize handler (writes to per-session storage, not instance fields)
  • Mcp-Session-Id echo on the HTTP transport
  • The standalone GET SSE endpoint and logging/setLevel handler
  • SSE-push for elicitation/sampling when the negotiated version is pre-2026-06

…while also implementing server/discover, subscriptions/listen, reading _meta per-request, and emitting IncompleteResult for 2026-06 clients. The InitializeRequest/InitializeResult types remain in schema.ts for exactly this purpose.

Clients supporting both server generations

The spec gives the probe sequence (differs by transport):

  • HTTP: "Try a modern request directly. If the server returns 400 Bad Request (or any other version error indicating the server does not implement the modern protocol), fall back to initialize and continue with the legacy version for subsequent requests."
  • stdio: "Probe with server/discover first, setting its preferred modern version in _meta. If the server returns Method not found (-32601), fall back to the legacy initialize handshake. If the server returns UnsupportedProtocolVersionError, the server speaks a version of MCP without initialize — use one of its advertised supportedVersions instead of falling back."

A client that only supports 2026-06 doesn't probe — it sends its version and handles UnsupportedProtocolVersionError normally.

Scope of the "MUST NOT infer" clause

lifecycle.mdx:19 says "Servers MUST NOT rely on prior requests over the same connection to establish context." This is a 2026-06-mode constraint (it's in the draft spec describing how _meta works). Reading capabilities from per-session storage for a legacy client that did initialize is not "inferring from prior requests" — it's honoring the 2025-11 protocol's handshake. The dual-support MAY would be self-contradictory otherwise.

The discriminator: if a request has _meta.clientCapabilities, use it and don't fall back. If absent (legacy client), per-session lookup is the 2025-11 behavior. The violation would be reading session storage as a fallback for a request that *did* send _meta.

What this means for the SDK split

Legacy support is a removable module (SessionCompat / BackchannelCompat in the rebuild), not version-gates scattered through core. Wired = the server is dual-support and the spec's MAY is satisfied. Unwired = pure 2026-06, and initialize/setLevel/standalone-GET return -32601 — which is the spec-correct response for a modern-only server.
