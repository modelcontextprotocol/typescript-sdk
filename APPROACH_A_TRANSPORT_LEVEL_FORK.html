Approach A: Transport-Level Fork ("Routing Transport")

Date: 2026-06-30  |  Status: Design exploration  |  Companion: transport-changes.html, Approach B

Core idea: Swap the transport class — everything else stays the same. On the server side, HTTPVersionRoutingTransport replaces NodeStreamableHTTPServerTransport. On the client side, VersionProbingHTTPClientTransport replaces StreamableHTTPClientTransport. McpServer, Client, tool handlers, and connect() are all unchanged. Version detection, protocol adaptation, and MRTR handling happen inside the transport layer.

0. User-Facing API: End-to-End Example

A minimal example: one server with a greet tool, one client calling it. Shows exactly what application code looks like for both protocol versions under Approach A.

0.1 Today's API (for reference — 2025-11 only)

This is the current SDK API. Shown for comparison so you can see exactly what changes.

How server.connect() works: Today, connect() already passes information from the server to the transport (onmessage, setSupportedProtocolVersions). The routing transport extends this — during connect(), McpServer detects it's a routing transport and also passes the handler registry (tools, resources, prompts, server info, capabilities). The transport uses this to create per-session legacy server instances and a single modern server instance internally. No new concepts for the user.

Under the hood (HTTP): The routing transport absorbs the session map boilerplate that users currently write manually in simpleStreamableHttp.ts: it creates a WebStandardStreamableHTTPServerTransport + legacy protocol handler per legacy session, and a single modern transport + modern protocol handler for all 2026-06 requests.

Under the hood (stdio): The routing transport detects the version from the first message, creates the appropriate legacy or modern protocol handler, and locks for the connection lifetime.

0.3 Approach A: Client connecting (both protocol versions)

Same pattern as server side: swap the transport class. The Client class is unchanged. A VersionProbingHTTPClientTransport replaces StreamableHTTPClientTransport and handles version detection, _meta injection, and MRTR retry loops internally. 
import { Client } from '@modelcontextprotocol/client';
import { VersionProbingHTTPClientTransport } from '@modelcontextprotocol/client';

// 1. Client class is UNCHANGED
const client = new Client({ name: 'my-client', version: '1.0.0' });

// 2. Swap the transport class (only change)
const transport = new VersionProbingHTTPClientTransport(
  new URL('http://localhost:3000/mcp')
);

// 3. connect() is UNCHANGED — transport handles version probing internally:
//    - Probes server/discover first
//    - If server responds: modern mode (inject _meta, handle MRTR)
//    - If -32601 or 400: legacy mode (initialize handshake, pass-through)
await client.connect(transport);

// 4. callTool() is UNCHANGED — works regardless of server version
const result = await client.callTool({ name: 'greet', arguments: { name: 'World' } });
console.log(result.content[0].text); // "Hello, World!"

await transport.close();
What the transport does internally:
  • Version probing: During start(), probes server/discover. If the server responds, switches to modern mode. If -32601 or 400, stays in legacy mode.
  • Handshake interception: In modern mode, when Client.connect() tries to send initialize, the transport intercepts it and sends server/discover instead, translating the response back into InitializeResult shape. The Client is unaware.
  • _meta injection: In modern mode, every outgoing request automatically gets _meta.protocolVersion, _meta.clientCapabilities, _meta.clientInfo, plus the Mcp-Method header.
  • MRTR retry loop: If the response is IncompleteResult, the transport resolves the input requests (using registered elicitation/sampling handlers), retries automatically, and returns only the final CallToolResult.

2. Version Detection Algorithm

For HTTP, detection is a single header check with no ambiguity. The 2026-06 spec makes Mcp-Method a required header on every request. This header did not exist in 2025-11. Its presence or absence is the definitive signal — no body parsing needed.

flowchart TD
    START(["Incoming HTTP request"]) --> H1{"Mcp-Method
header present?"}
    H1 -->|"Yes"| MODERN["Route to modern path"]
    H1 -->|"No"| LEGACY["Route to legacy path"]

    style MODERN fill:#d1fae5,stroke:#16a34a
    style LEGACY fill:#fef3c7,stroke:#f59e0b

2.1 Why there is no ambiguity

Header2025-112026-06Conclusion
Mcp-MethodDoes not exist in the specRequired on every request (SEP-2243)Presence = modern, absence = legacy. Sufficient on its own.
MCP-Protocol-VersionOptional / advisoryRequired; must match _meta.protocolVersionRedundant confirmation. Not needed for routing.
Mcp-Session-IdPresent (after init)Does not exist (sessions removed)Also sufficient, but only after the first request.

Even the first request from a 2025-11 client (the initialize request, which has no Mcp-Session-Id yet) is unambiguous: it won't have Mcp-Method either, because that header doesn't exist in 2025-11.

Performance: Version detection is always O(1) — a single header lookup on every request, including the first. No body parsing, no fallback logic, no ambiguous cases.

2.2 TypeScript sketch

type ProtocolGeneration = 'legacy' | 'modern';

function detectVersion(req: Request): ProtocolGeneration {
  // Mcp-Method is required in 2026-06, does not exist in 2025-11.
  // This single check is sufficient for all requests, including
  // the first initialize request from a legacy client.
  return req.headers.has('mcp-method') ? 'modern' : 'legacy';
}

2.3 stdio detection (body-based)

stdio has no HTTP headers, so detection falls back to inspecting the first message body. This only happens once (stdio locks on first message).

function detectVersionFromBody(msg: JSONRPCMessage): ProtocolGeneration {
  if (isJSONRPCRequest(msg)) {
    if (msg.method === 'initialize') return 'legacy';
    if (msg.params?._meta?.protocolVersion) return 'modern';
  }
  return 'legacy'; // default for backward compat
}

Design decision: stdio locks on the first message and never re-evaluates. A single client connects via stdio; mid-connection version switching is explicitly forbidden. If the host process wants a different protocol version, it should spawn a new server process.

4.1 Implementation sketch

class VersionRoutingStdioTransport {
  private inner: StdioServerTransport;
  private lockedVersion: ProtocolGeneration | null = null;
  private legacyHandler: Server;      // existing Protocol+Server (frozen)
  private modernHandler: ModernProtocolHandler;

  constructor(registry: HandlerRegistry, stdin?: Readable, stdout?: Writable) {
    this.inner = new StdioServerTransport(stdin, stdout);
  }

  async start(): Promise<void> {
    // Intercept onmessage to detect version on first message
    this.inner.onmessage = (msg, extra) => {
      if (this.lockedVersion === null) {
        this.lockedVersion = detectVersionFromBody(msg);
        // Connect the correct protocol handler to the inner transport
        const handler = this.lockedVersion === 'legacy'
          ? this.legacyHandler : this.modernHandler;
        // Wire onmessage to the selected protocol handler
        this.inner.onmessage = handler.handleMessage.bind(handler);
      }
      // Forward this first message too
      this.inner.onmessage!(msg, extra);
    };
    await this.inner.start();
  }
}

5.1 Transport implementation sketch

class VersionProbingHTTPClientTransport implements Transport {
  private mode: 'unknown' | 'legacy' | 'modern' = 'unknown';
  private inner: StreamableHTTPClientTransport;
  private serverCapabilities?: ServerCapabilities;

  constructor(url: URL, options?: ClientTransportOptions) {
    this.inner = new StreamableHTTPClientTransport(url, options);
  }

  async start(): Promise<void> {
    await this.inner.start();

    // Probe: try server/discover
    try {
      const result = await this.sendProbe();
      this.mode = 'modern';
      this.serverCapabilities = result.capabilities;
    } catch (err) {
      if (isMethodNotFound(err) || isBadRequest(err)) {
        this.mode = 'legacy'; // Server is 2025-11
      } else if (isUnsupportedVersionError(err)) {
        // Server is modern but different version
        this.mode = 'modern';
        // Could retry with a supported version
      } else {
        throw err;
      }
    }
  }

  async send(message: JSONRPCMessage, options?: TransportSendOptions): Promise<void> {
    if (this.mode === 'modern') {
      // Intercept initialize → translate to server/discover response
      if (isInitializeRequest(message)) {
        this.fakeInitializeResponse(message);
        return;
      }
      // Inject _meta on every outgoing request
      this.injectMeta(message);
    }
    // Legacy mode: pass through unchanged
    await this.inner.send(message, options);
  }

  // When a response arrives, check for IncompleteResult (MRTR)
  set onmessage(handler: Transport['onmessage']) {
    this.inner.onmessage = (msg, extra) => {
      if (this.mode === 'modern' && isIncompleteResult(msg)) {
        // Resolve inputs, retry, return final result
        this.handleMRTR(msg, handler);
        return;
      }
      handler?.(msg, extra);
    };
  }
}

6.2 How connect() passes the registry

// Protocol already has this Map — just add a getter:
class Protocol {
  private _requestHandlers: Map<string, Handler> = new Map();

  getRequestHandlers(): ReadonlyMap<string, Handler> {
    return this._requestHandlers;
  }
}

// In Protocol.connect() — works for both Client and Server:
async connect(transport: Transport): Promise<void> {
  this._transport = transport;

  // Existing: set callbacks
  transport.onmessage = (msg, extra) => { /* ... */ };
  transport.setSupportedProtocolVersions?.(this._supportedProtocolVersions);

  // NEW: pass protocol config to routing transports
  transport.setProtocolConfig?.({
    requestHandlers: this._requestHandlers,  // live reference, same Map
    serverInfo: this._serverInfo,            // undefined on Client
    capabilities: this._capabilities,
  });

  await this._transport.start();
}
Why this works for both sides:
  • Server transport receives handlers like tools/call, resources/read, prompts/get. Uses them to dispatch requests to per-session protocol handlers, and to respond to server/discover with capabilities.
  • Client transport receives handlers like elicitation/create, sampling/createMessage. Uses them to resolve MRTR inputRequests when the server returns IncompleteResult.
  • The Map is a live reference — handlers registered after connect() are immediately visible. No copy, no sync needed.
  • setProtocolConfig is an optional method on the Transport interface. Regular transports ignore it. Only routing/probing transports implement it.

6.3 How the client transport resolves MRTR

// Inside VersionProbingHTTPClientTransport:
private async resolveMRTR(
  incompleteResult: IncompleteResult,
  originalRequest: JSONRPCRequest,
): Promise<Result> {
  const inputResponses: InputResponse[] = [];

  for (const inputReq of incompleteResult.inputRequests) {
    // Look up the handler from the registry passed during connect()
    const handler = this.requestHandlers.get(inputReq.type);
    if (!handler) throw new Error(`No handler for input type: ${inputReq.type}`);

    // Call it directly — same handler the Client registered
    const result = await handler(
      { method: inputReq.type, params: inputReq.params } as JSONRPCRequest,
      { /* minimal context: signal, etc. */ }
    );
    inputResponses.push({ key: inputReq.key, result });
  }

  // Retry the original request with inputResponses
  const retryParams = {
    ...originalRequest.params,
    requestState: incompleteResult.requestState,
    inputResponses,
  };
  // Send retry via inner transport, may get another IncompleteResult (loop)
  return this.sendAndMaybeRetry({ ...originalRequest, params: retryParams });
}

7. MRTR Transparency

Key challenge: In 2026-06, server→client requests (elicitation, sampling) are replaced by MRTR (Multi-Round Tool Resolution). The server cannot push requests to the client. Instead, it returns IncompleteResult with inputRequests, and the client retries with inputResponses. Making this transparent to handler authors is the thorniest design problem.

7.3 Implementation sketch

class InputRequiredException extends Error {
  constructor(
    public readonly inputRequests: InputRequest[],
    public readonly requestState: unknown,
  ) {
    super('Handler requires client input (MRTR flow)');
  }
}

// Modern adapter's elicitInput implementation
function createModernElicitInput(
  inputResponses: InputResponse[] | undefined,
  requestState: unknown,
): (params: ElicitRequestParams) => Promise<ElicitResult> {
  let callIndex = 0;

  return async (params: ElicitRequestParams): Promise<ElicitResult> => {
    const currentIndex = callIndex++;

    // Check if this elicitation was resolved in a prior round
    if (inputResponses && currentIndex < inputResponses.length) {
      return inputResponses[currentIndex].result as ElicitResult;
    }

    // Not resolved yet: throw to trigger IncompleteResult
    throw new InputRequiredException(
      [{ type: 'elicitation', params }],
      requestState,
    );
  };
}

// In the modern dispatch layer:
async function dispatchModern(
  method: string, params: any, registry: HandlerRegistry
): Promise<Result> {
  try {
    const result = await executeHandler(method, params, registry);
    return { result_type: 'complete', ...result };
  } catch (err) {
    if (err instanceof InputRequiredException) {
      return {
        result_type: 'input_required',
        inputRequests: err.inputRequests,
        requestState: err.requestState,
      };
    }
    throw err;
  }
}

For tools that do, the guidance is: "your handler may be called multiple times for a single logical request. Use requestState if you have side effects before the elicitation call."

8. Legacy Code Handling Strategies

Both Protocol/Server and the existing transport code need a handling strategy. Here are three options with trade-offs specific to Approach A:

8.1 Literally frozen

DimensionAssessment
Risk to legacy clientsZero. No code changes, no regressions possible.
Review burdenNone. Frozen code needs no review.
HandlerRegistry extractionCannot extract from McpServer without changing it. Must build a separate shim that reads McpServer's internal state or duplicates registrations.
Interface fitThe existing Server.connect(transport) contract works as-is with the inner legacy transport. But McpServer's lazy setRequestHandler installation means the registry concept must be built around it, not inside it.
Approach A compatibilityGood. The inner legacy transport connects to the frozen Server normally. The routing transport wraps it without any changes to legacy code.

How it works: The VersionRoutingTransport wraps the existing WebStandardStreamableHTTPServerTransport as a black box. Tool handlers are registered via a new HandlerRegistryShim that intercepts McpServer registrations and mirrors them to the modern server.

8.2 Refactor for interface fit

DimensionAssessment
HandlerRegistry extractionClean extraction. McpServer's _registeredTools etc. move into a shared HandlerRegistry class. Both servers read from it.
Code organizationBetter separation of concerns. Protocol/Server/McpServer each have clearer responsibilities.
Risk to legacy clientsLow-moderate. Internal restructuring could introduce subtle bugs. Mitigated by comprehensive test suite.
Review burdenModerate. Refactoring changes need careful review to ensure behavioral equivalence.
Approach A compatibilityNatural fit. The refactored Server accepts a HandlerRegistry in its constructor, making the shared registry explicit.

How it works: Extract _registeredTools, _registeredResources, _registeredPrompts and the associated dispatch logic from McpServer into a standalone HandlerRegistry class. McpServer becomes a thin facade. Both the legacy and modern protocol handlers (internal to the routing transport) read from the same HandlerRegistry instance.

8.3 Copy and freeze

DimensionAssessment
Risk to legacy clientsZero. Frozen copy is a snapshot.
IndependenceMaximum. Legacy and modern evolve completely independently.
Code duplicationHigh. Protocol, Server, McpServer, transport — all duplicated. Bug fixes must be applied in two places.
Handler sharingRequires a compatibility layer between the frozen copy's handler types and the modern handler types. Type drift over time.
Approach A compatibilityWorks but wasteful. Two copies of transport, protocol, server when the routing transport already provides isolation.

How it works: Fork the current Protocol, Server, McpServer, and transport code into a legacy/ directory. The original files evolve freely for 2026-06. The frozen copy serves legacy clients until 2025-11 is deprecated.

8.4 Recommendation for Approach A

Recommended: Refactor for interface fit. The HandlerRegistry extraction is essential for the shared-handler architecture. Without it, handler registration requires either duplication or a brittle shim. The refactoring risk is manageable with the existing test suite, and the result is cleaner code organization that serves both protocol versions naturally.

12. Open Issues from Architecture Review

Issues identified by Software Architect and Backend Architect reviews. Items marked RESOLVED have solutions; items marked OPEN need further design work.

12.1 RESOLVED — MRTR handler access on client side

Problem: The client transport needs to resolve MRTR inputRequests but has no access to the Client's registered elicitation/sampling handlers.

Resolution: Reuse the same handler registry mechanism as the server side. Protocol already has _requestHandlers. During connect(), pass a live reference to the transport via setProtocolConfig(). See Section 6.

12.2 RESOLVED — connect() registry passing mechanism

Problem: How does the transport receive the handler registry without polluting the Transport interface?

Resolution: Add an optional setProtocolConfig?(config) method to the Transport interface. Protocol.connect() calls it if present. Regular transports ignore it. Same mechanism for both client and server. See Section 6.2.

12.3 RESOLVED — fakeInitializeResponse timing

Problem: The client transport intercepts initialize in send() and returns early. But Client.connect() calls this.request() which waits for a response via _responseHandlers. Returning early from send() causes the Client to hang.

Resolution: The transport must synthesize a JSON-RPC response and deliver it asynchronously via onmessage:

async send(message: JSONRPCMessage): Promise<void> {
  if (this.mode === 'modern' && isInitializeRequest(message)) {
    // Synthesize InitializeResult from cached DiscoverResult
    const fakeResponse: JSONRPCResponse = {
      jsonrpc: '2.0',
      id: (message as JSONRPCRequest).id,
      result: {
        protocolVersion: this.negotiatedVersion,
        capabilities: this.cachedDiscoverResult.capabilities,
        serverInfo: this.cachedDiscoverResult.serverInfo,
        instructions: this.cachedDiscoverResult.instructions,
      },
    };
    // Deliver asynchronously to avoid re-entrance
    queueMicrotask(() => this.onmessage?.(fakeResponse));
    return;
  }
  // ...
}

12.4 OPEN — Session lifecycle management

Problem: The routing transport creates per-session legacy stacks (~50-100KB each). No cleanup mechanism is specified for abandoned sessions (client disconnects without sending DELETE).

Required:

  • Session-to-inner-transport map with configurable idle TTL
  • Maximum concurrent session limit (reject new initialize when full)
  • DELETE request routing to correct inner transport + cleanup
  • Periodic sweep for timed-out sessions
  • onsessionclosed callback for both explicit DELETE and idle timeout

12.5 OPEN — Notification routing for modern path

Problem: In 2025-11, list_changed and logging push via SSE. In 2026-06, these move to subscriptions/listen (opt-in) and in-band logging (per-request). The design does not specify how the modern protocol handler delivers these.

Required:

  • Modern path: implement subscriptions/listen as a long-lived POST with SSE response
  • In-band logging: log() appends to the current request's response stream, only if _meta.logLevel was set
  • Progress: notifications/progress is in-band on the request stream (unchanged but scoping matters)
  • When a tool is dynamically added, legacy sessions get a list_changed push; modern subscribers get notified via their subscriptions/listen stream

12.6 OPEN — MRTR deterministic elicitation ordering

Problem: The call-index scheme (Section 7.3) assumes handlers call elicitInput() in the same deterministic order on every replay. Data-dependent branching can break this.

Options:

  • Named keys: Use inputRequests[].key / inputResponses[].key instead of positional indexing. Handlers provide a stable key per elicitation call.
  • Document as constraint: "Handlers with data-dependent branching between elicitation calls are not safely supported under MRTR replay. Use requestState to ensure deterministic ordering."

12.7 OPEN — GET/DELETE routing and standalone SSE

Problem: The routing transport handles POST requests. But legacy clients also use GET (standalone SSE stream) and DELETE (session termination). These must be routed to the correct inner transport by session ID.

12.8 Complexity estimate

Both reviewers noted the "~200 lines" estimate for the routing transport is too low. Realistic estimate with session management, GET/DELETE routing, and lifecycle: 400-600 lines for the HTTP routing transport, plus ~150 lines for the stdio variant.

13. Summary

Approach A: swap the transport class, everything else stays the same.

  • User-facing API: McpServer, Client, tool handlers, and connect() are unchanged. The only change is the transport class import.
  • Server side: HTTPVersionRoutingTransport detects version via Mcp-Method header (O(1), no ambiguity), manages per-session legacy stacks and a single modern handler internally.
  • Client side: VersionProbingHTTPClientTransport probes server/discover, intercepts handshakes, injects _meta, resolves MRTR via the shared handler registry.
  • Handler registry: Protocol._requestHandlers is exposed via setProtocolConfig() during connect() — the same mechanism and same Map type on both client and server.
  • MRTR: Exception-based continuation on the server side; direct handler invocation from the registry on the client side. Transparent for the 95% of tools that don't use elicitation/sampling.

Validated by review: Version detection, per-session legacy stacks, stdio locking, clean removal path, handler registry approach.

Open items: Session lifecycle management (TTL, cleanup), notification routing for modern path (subscriptions/listen, in-band logging), GET/DELETE routing, MRTR deterministic ordering constraint.

Estimated complexity: ~400-600 lines for HTTP routing transport, ~150 lines for stdio variant.
