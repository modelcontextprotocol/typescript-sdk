Transport Flip: McpVersionRouter Architecture Proposal

Goal: Separate three concerns that are currently entangled in the SDK: wire transport, protocol version routing, and handler registration. Make the codebase stateless-first for 2026-06 while preserving legacy 2025-11 support via a frozen, routable code path.

Scope: Server-side architecture (the client-side story is simpler and follows the same principles). MRTR is conceptually accounted for but not deeply designed here.

Based on: transport-changes.html (spec diff), SEPs 2575, 2567, 2322, 2260, 2243

1. The Problem: Current Architecture Is Stateful-First

1.3 The core tension

2025-11 model: "Handshake establishes state, requests are lightweight."
2026-06 model: "Every request is self-describing, no connection state."
The SDK must support both on one server instance.

The existing class hierarchy makes the 2025-11 model the only path. Every request flows through Protocol, which assumes a connected transport with session state. Adding 2026-06 support by bolting it onto Protocol would mean either:

  • Option A: Scatter version-gates throughout Protocol/Server (maintenance nightmare, violates single-responsibility)
  • Option B: Refactor Protocol to be stateless-first (massive blast radius, breaks existing code)
  • Option C: (This proposal) Separate the concerns, freeze Protocol/Server for legacy, build a clean modern path alongside

2. Proposed Architecture: Three Separated Concerns

Layer 1: McpServer — Handler Registry both eras

Pure definition of what the server can do. Tools, resources, prompts, capabilities, server info. No transport, no connection state, no protocol framing. Both the modern and legacy paths read from this same registry.

Layer 2: McpVersionRouter — Protocol Version Routing both eras

An abstract class / interface with transport-specific implementations. The classification logic differs by transport: HTTP uses Mcp-Method / MCP-Protocol-Version headers; stdio checks the first message method; WebSocket/gRPC may use their own metadata. The SDK ships default implementations for standard transports (HttpVersionRouter, StdioVersionRouter). Custom transport authors can implement their own router for transport-specific routing signals.

For modern (2026-06): builds per-request context from _meta, dispatches to McpServer handlers directly. For legacy (2025-11): delegates to a frozen Protocol/Server instance. Handles server/discover, version negotiation, and is the future hook point for MRTR and subscriptions/listen.

Layer 3: Transport — Wire Protocol both eras

How bytes move. HTTP request/response, stdio stdin/stdout, WebSocket frames, gRPC streams. Message framing, serialization, connection lifecycle. Transport and router form a coupled pair — the transport knows what metadata is available for routing, the router knows how to interpret it. Custom transport authors implement their Transport and optionally a matching Router. A transport can support legacy, modern, or both eras.

2.2 McpServer changes: handler registry, not server runner

Today, McpServer pushes handlers INTO Server (which stores them in Protocol._requestHandlers). The handlers are trapped inside the Protocol/Server chain. In the new model, McpServer holds handlers independently and exposes them for consumption:

// New McpServer surface (additions shown)
class McpServer {
    // === UNCHANGED: same registration API ===
    registerTool(name, config, handler): RegisteredTool;
    registerResource(name, uri, config, handler): RegisteredResource;
    registerPrompt(name, config, handler): RegisteredPrompt;

    // === NEW: handler registry access ===
    // Capabilities derived from registrations
    getCapabilities(): ServerCapabilities;
    getServerInfo(): Implementation;
    getInstructions(): string | undefined;

    // Dispatch a request to the appropriate handler
    // Called by McpVersionRouter for modern requests
    dispatch(method: string, request: JSONRPCRequest, ctx: ServerContext): Promise<Result>;

    // Enumerate handlers (for legacy bridge to populate frozen Server)
    getHandlerEntries(): Iterable<[string, RequestHandler]>;

    // === REMOVED: no longer on McpServer ===
    // connect(transport)  -- moved to McpVersionRouter
    // server: Server      -- no longer exposed; internal detail of legacy path
}

Key insight: McpServer.dispatch() is the modern path. No Protocol, no transport, no session. Just: method → handler → result. The McpVersionRouter builds the ServerContext from _meta before calling dispatch.

2.3 McpVersionRouter: abstract class with transport-specific subclasses

The router is abstract. Each transport type provides its own subclass that knows how to read transport-specific routing signals. The base class provides the shared modern dispatch, legacy bridge, server/discover, and future MRTR/subscription infrastructure.

type McpEra = 'legacy' | 'modern';

// Transport-specific metadata passed alongside messages
interface TransportMeta {
    // HTTP-specific
    httpHeaders?: Record<string, string>;
    httpMethod?: string;  // GET, POST, DELETE
    // Auth
    authInfo?: AuthInfo;
    // Connection identity (for per-connection era locking on persistent transports)
    connectionId?: string;
}

abstract class McpVersionRouter {
    constructor(
        protected mcpServer: McpServer,
        protected options?: {
            // Enable legacy (2025-11) support.
            // When true, creates internal frozen Server/Protocol for legacy sessions.
            // When false (default for new code), initialize returns -32601, etc.
            legacySupport?: boolean;

            // Supported protocol versions (defaults to SUPPORTED_PROTOCOL_VERSIONS)
            supportedVersions?: string[];
        }
    );

    // === ABSTRACT: transport-specific classification ===
    // Each subclass overrides this to inspect transport-relevant signals.
    abstract classify(message: JSONRPCMessage, meta?: TransportMeta): McpEra;

    // === CONCRETE: shared infrastructure ===
    // Modern path: build context from _meta, dispatch to McpServer
    protected handleModernRequest(request: JSONRPCRequest, meta?: TransportMeta): Promise<Result>;

    // Legacy path: route to frozen Server/Protocol via bridge
    protected handleLegacyMessage(message: JSONRPCMessage, meta?: TransportMeta): void;

    // server/discover: always modern, reads from McpServer
    protected handleDiscover(): DiscoverResult;

    // For persistent transports (stdio, WebSocket, etc.)
    serve(transport: Transport): Promise<void>;

    // Shutdown
    close(): Promise<void>;
}

2.4 Shared HandlerRegistry: the link between modern and legacy

A key refinement over "fully frozen Protocol": instead of copying handlers into each legacy Server instance, we extract Protocol's handler maps into a shared HandlerRegistry that both paths read from. This is a minimal, targeted change to Protocol — the maps move from private fields to an external object. Everything else stays the same.

What gets extracted (handler definitions — shared)

// Extracted from Protocol's private fields into a standalone object.
// Holds handler DEFINITIONS (method → handler function).
// Both modern dispatch and legacy Protocol._onrequest read from this.
class HandlerRegistry<ContextT> {
    private _requestHandlers = new Map<string, RequestHandler<ContextT>>();
    private _notificationHandlers = new Map<string, NotificationHandler>();

    getRequestHandler(method: string): RequestHandler<ContextT> | undefined;
    setRequestHandler(method: string, handler: RequestHandler<ContextT>): void;
    removeRequestHandler(method: string): void;
    hasRequestHandler(method: string): boolean;

    getNotificationHandler(method: string): NotificationHandler | undefined;
    setNotificationHandler(method: string, handler: NotificationHandler): void;
    removeNotificationHandler(method: string): void;
}

What stays on Protocol (per-request/per-connection state — not shared)

// These maps are per-Protocol-instance. They track in-flight requests
// on a specific connection, not handler definitions. NOT extracted.
class Protocol<ContextT> {
    private _responseHandlers: Map<number, ResponseHandler>;   // outbound request correlation
    private _progressHandlers: Map<number, ProgressCallback>;  // outbound progress tracking
    private _requestHandlerAbortControllers: Map<RequestId, AbortController>; // per-request cancellation
    private _timeoutInfo: Map<number, TimeoutInfo>;            // per-request timeouts
}

Protocol change (minimal)

class Protocol<ContextT> {
    private _registry: HandlerRegistry<ContextT>;

    // NEW: accepts external registry. Creates its own if not provided (backwards compat).
    constructor(options?: { registry?: HandlerRegistry<ContextT>; /* ...existing options */ }) {
        this._registry = options?.registry ?? new HandlerRegistry();
    }

    // CHANGED: delegates to registry. Capability check + wrapping still happen here.
    setRequestHandler(method, handler) {
        this.assertRequestHandlerCapability(method);
        this._registry.setRequestHandler(method, this._wrapHandler(method, handler));
    }

    // CHANGED: reads from registry instead of private map.
    private _onrequest(request, extra) {
        const handler = this._registry.getRequestHandler(request.method)
            ?? this.fallbackRequestHandler;
        // ... rest of _onrequest is UNCHANGED
    }

    // UNCHANGED: setNotificationHandler, removeRequestHandler, etc.
    // all delegate to this._registry instead of this._notificationHandlers / this._requestHandlers

    // UNCHANGED: _responseHandlers, _progressHandlers, _requestHandlerAbortControllers
    // stay as private fields on Protocol (per-connection state)
}

How McpServer uses it

class McpServer {
    // McpServer creates the single shared registry
    private _registry = new HandlerRegistry<ServerContext>();

    // Compound handlers (tools/list, tools/call, etc.) are registered in the shared registry.
    // These closures read from _registeredTools / _registeredResources / _registeredPrompts
    // at DISPATCH time, not at registration time — so tool additions/removals
    // are automatically visible to all consumers without re-registration.
    private setToolRequestHandlers() {
        this._registry.setRequestHandler('tools/list', async (request, ctx) => {
            // reads this._registeredTools HERE, at call time
            return { tools: Object.entries(this._registeredTools).map(...) };
        });
        this._registry.setRequestHandler('tools/call', async (request, ctx) => {
            // reads this._registeredTools[name] HERE, at call time
            const tool = this._registeredTools[request.params.name];
            return tool.executor(request.params.arguments, ctx);
        });
    }

    // Modern path: router reads handlers directly from the shared registry
    get registry(): HandlerRegistry<ServerContext> { return this._registry; }

    // Legacy path: Server constructed WITH the shared registry
    // → Protocol._onrequest reads from the same registry
    // → handler mutations are instantly visible, no event subscription needed
}

Why this works: McpServer's compound handlers (e.g., the tools/call handler registered via setToolRequestHandlers()) already read from _registeredTools at dispatch time, not at registration time. The registry holds the compound handler once. When registerTool() adds a tool to _registeredTools, the next invocation of the compound handler sees it — regardless of whether the caller is the modern dispatch path or a legacy Protocol._onrequest.

What this eliminates:

  • No getHandlerEntries() needed — consumers share the registry directly
  • No handler population step in the legacy bridge — Server instances are constructed with the shared registry
  • No stale handlers — mutations are instantly visible to all consumers
  • No event subscription / change propagation — the shared reference is the propagation mechanism

Blast radius: Protocol gains one constructor option (registry?) and reads from this._registry instead of this._requestHandlers. Method signatures, behavior, error handling, timeout machinery, cancellation — all unchanged. Existing code that creates Protocol/Server without passing a registry gets the same behavior as today (auto-created internal registry).

5. The Frozen Legacy Bridge

The legacy bridge connects the McpVersionRouter to frozen Server/Protocol instances for 2025-11 sessions. With the shared HandlerRegistry (§2.4), the bridge is simpler than it would otherwise be — it handles per-session state and transport multiplexing only, not handler management.

5.1 What the bridge provides

ConcernHandled by
Handler lookup (tools/call → handler)Shared HandlerRegistry — Server's Protocol reads from the same registry as the modern path. No handler copying or synchronization.
Per-session state (_clientCapabilities, _clientVersion, _loggingLevels)Frozen Server instance per session — set during initialize, read during requests. Unchanged from today.
Transport ownership (Protocol expects exclusive onmessage/send)BridgeTransport — thin in-memory pipe. Router injects messages in, Protocol's outgoing messages come back out. Necessary because Protocol.connect() takes exclusive ownership of its transport.
Bidirectional requests (server → client: elicitation, sampling)BridgeTransport relays outgoing requests from Protocol to the router, which forwards to the real transport. Client's response comes back the same path. Protocol's timeout/cancellation machinery works unchanged.

5.2 Implementation

// Internal to McpVersionRouter - not public API
class LegacyBridge {
    private sessions: Map<string | undefined, { server: Server; bridge: BridgeTransport }>;

    constructor(
        private registry: HandlerRegistry<ServerContext>,  // shared with McpServer
        private serverInfo: Implementation,
        private serverOptions: ServerOptions,
    ) {}

    getOrCreate(sessionId: string | undefined): { server: Server; bridge: BridgeTransport } {
        let entry = this.sessions.get(sessionId);
        if (!entry) {
            // Server constructed WITH the shared registry.
            // Protocol._onrequest reads handlers from this registry —
            // same handlers as the modern path, always up to date.
            const server = new Server(this.serverInfo, {
                capabilities: this.serverOptions.capabilities,
                instructions: this.serverOptions.instructions,
                registry: this.registry,  // ← shared HandlerRegistry
            });

            // No handler population loop needed.
            // The shared registry already has all handlers.

            const bridge = new BridgeTransport();
            server.connect(bridge);

            entry = { server, bridge };
            this.sessions.set(sessionId, entry);
        }
        return entry;
    }

    // Inject a message into the legacy Server as if it came from the transport
    async injectMessage(sessionId: string | undefined,
                        message: JSONRPCMessage,
                        extra?: MessageExtraInfo) {
        const { bridge } = this.getOrCreate(sessionId);
        bridge.injectIncoming(message, extra);
    }

    // Receive outgoing messages from the legacy Server
    // (bidirectional: Protocol.send() → BridgeTransport.send() → this callback)
    onOutgoing(callback: (sessionId: string | undefined,
                          message: JSONRPCMessage) => void) {
        // BridgeTransport.send() calls this callback
    }

    // Cleanup
    destroySession(sessionId: string | undefined): void {
        const entry = this.sessions.get(sessionId);
        if (entry) {
            entry.server.close();
            this.sessions.delete(sessionId);
        }
    }
}

5.3 BridgeTransport: the in-memory pipe

// Thin adapter that Protocol treats as its Transport.
// No serialization, no network I/O — messages pass by reference.
class BridgeTransport implements Transport {
    // Router calls this to deliver messages TO Protocol
    injectIncoming(message: JSONRPCMessage, extra?: MessageExtraInfo) {
        this.onmessage?.(message, extra);  // Protocol installed this in connect()
    }

    // Protocol calls this to send messages OUT (responses + bidirectional requests)
    async send(message: JSONRPCMessage) {
        this.onOutgoing?.(message);        // Router installed this
    }

    async start() { /* no-op */ }
    async close() { this.onclose?.(); }

    // Protocol sets these during connect()
    onmessage?: (msg: JSONRPCMessage, extra?: MessageExtraInfo) => void;
    onclose?: () => void;
    onerror?: (error: Error) => void;

    // Router sets this to receive outgoing messages
    onOutgoing?: (message: JSONRPCMessage) => void;
}

Why a bridge transport? Protocol.connect() takes exclusive ownership of a Transport's onmessage/onclose/onerror callbacks. We can't share the real transport with Protocol. The bridge is a thin in-memory pipe that relays messages between the router and the frozen Protocol/Server. Messages pass by reference — no serialization, no copying.

5.4 Bidirectional round-trip through the bridge

In legacy mode, the Server can initiate requests to the client (elicitation, sampling, roots). These flow through the bridge as outgoing messages:

Key property: Protocol's existing timeout and cancellation machinery works unchanged through the bridge. If the client disconnects, the router calls bridge.onerror() or bridge.onclose(), and Protocol rejects all pending response handlers. The bridge is transparent to Protocol's error model.

6. MRTR: Conceptual Hooks (Future)

MRTR (SEP-2322) replaces server-to-client push (elicitation, sampling) with a return-and-retry pattern. We don't implement it now, but the architecture must accommodate it.

6.1 How MRTR fits in the modern path

sequenceDiagram
    participant C as Client
    participant R as McpVersionRouter
    participant M as McpServer

    C->>R: tools/call with _meta
    R->>M: dispatch tools/call request ctx
    M->>M: Handler needs user input
    M-->>R: IncompleteResult result_type input_required with inputRequests and requestState
    R-->>C: IncompleteResult

    Note over C: Client resolves inputRequests locally, shows elicitation UI to user

    C->>R: tools/call with _meta, requestState, inputResponses
    R->>M: dispatch tools/call request ctx with requestState + inputResponses
    M->>M: Handler resumes with user input
    M-->>R: CallToolResult result_type complete
    R-->>C: CallToolResult

Where MRTR state lives: In requestState (opaque token in the request/response). Not in the server, not in the router, not in the transport. Fully stateless.

How it differs from legacy: In legacy mode, the server pushes elicitation/create over the SSE stream and the client responds on the same connection. This is handled by the frozen Protocol/Server's existing bidirectional request flow. The modern MRTR path is return-and-retry, with no push.

6.2 Architecture implications

  • McpServer.dispatch() returns Result, which can be CallToolResult OR IncompleteResult OR CreateTaskResult. The router inspects result_type to determine response handling.
  • The router could offer a convenience MRTR loop for transports that want automatic retry handling.
  • No new state is needed in McpServer or McpVersionRouter — requestState round-trips through the client.

7. subscriptions/listen: Modern Push Channel

In 2026-06, subscriptions/listen replaces the standalone HTTP GET SSE stream. It's a POST that opens a long-lived response for server-to-client notifications.

sequenceDiagram
    participant C as Client
    participant R as McpVersionRouter
    participant M as McpServer

    C->>R: subscriptions/listen with _meta, toolsListChanged true, resourcesListChanged true
    R->>R: Open long-lived response stream
    R-->>C: notifications/subscriptions/acknowledged

    Note over R,M: McpServer emits list-changed events

    M->>R: onToolListChanged
    R-->>C: notifications/tools/list_changed on subscription stream

    M->>R: onResourceListChanged
    R-->>C: notifications/resources/list_changed on subscription stream

The McpVersionRouter manages the subscription lifecycle. McpServer just emits events (it already does this today). The router filters and routes them to the appropriate subscription streams.

7.1 Subscription data model

// A single subscriptions/listen stream from a client
interface Subscription {
    id: string;                            // server-generated subscriptionId
    types: {                               // what the client opted in to
        toolsListChanged?: boolean;
        promptsListChanged?: boolean;
        resourcesListChanged?: boolean;
        resourceSubscriptions?: string[];  // specific resource URIs
    };
    writer: SubscriptionWriter;            // writes JSON-RPC notifications to the response stream
    signal: AbortSignal;                   // fires on client disconnect (HTTP) or transport close (stdio)
    lastEventId?: string;                  // for resumability via EventStore
}

// SubscriptionWriter abstracts the output format.
// HTTP: writes SSE events on the long-lived POST response.
// Stdio: writes JSON-RPC notifications to stdout (interleaved with other messages).
interface SubscriptionWriter {
    write(notification: JSONRPCNotification): void;
    close(): void;
}

7.2 Router subscription management

// On McpVersionRouter (base class)
abstract class McpVersionRouter {
    private _subscriptions = new Map<string, Subscription>();

    // === Subscription lifecycle ===

    protected handleSubscriptionsListen(
        request: JSONRPCRequest,
        writer: SubscriptionWriter,
        signal: AbortSignal,
    ): void {
        const sub: Subscription = {
            id: generateSubscriptionId(),
            types: request.params,          // { toolsListChanged, resourcesListChanged, ... }
            writer,
            signal,
        };
        this._subscriptions.set(sub.id, sub);

        // Send acknowledgement (spec: notifications/subscriptions/acknowledged)
        writer.write({
            jsonrpc: '2.0',
            method: 'notifications/subscriptions/acknowledged',
            params: { _meta: { subscriptionId: sub.id } },
        });

        // Auto-cleanup on client disconnect
        signal.addEventListener('abort', () => {
            this._subscriptions.delete(sub.id);
            writer.close();
        }, { once: true });
    }

    // === Fan-out: McpServer events → matching subscriptions ===

    private _fanOut(filterKey: keyof Subscription['types'], notification: JSONRPCNotification) {
        for (const sub of this._subscriptions.values()) {
            if (sub.types[filterKey]) {
                // Tag notification with subscriptionId per spec
                notification.params = {
                    ...notification.params,
                    _meta: { ...notification.params?._meta, subscriptionId: sub.id },
                };
                sub.writer.write(notification);
            }
        }
    }

    // Wired to McpServer's existing event emitters during serve()
    protected connectMcpServerEvents(mcpServer: McpServer): void {
        mcpServer.onToolListChanged(() =>
            this._fanOut('toolsListChanged',
                { jsonrpc: '2.0', method: 'notifications/tools/list_changed' }));

        mcpServer.onPromptListChanged(() =>
            this._fanOut('promptsListChanged',
                { jsonrpc: '2.0', method: 'notifications/prompts/list_changed' }));

        mcpServer.onResourceListChanged(() =>
            this._fanOut('resourcesListChanged',
                { jsonrpc: '2.0', method: 'notifications/resources/list_changed' }));

        // Resource-specific subscriptions:
        mcpServer.onResourceUpdated((uri: string) => {
            for (const sub of this._subscriptions.values()) {
                if (sub.types.resourceSubscriptions?.includes(uri)) {
                    sub.writer.write({
                        jsonrpc: '2.0',
                        method: 'notifications/resources/updated',
                        params: { uri, _meta: { subscriptionId: sub.id } },
                    });
                }
            }
        });
    }

    // === Cleanup ===

    async close(): Promise<void> {
        for (const sub of this._subscriptions.values()) {
            sub.writer.close();
        }
        this._subscriptions.clear();
    }
}

7.5 Design decisions

DecisionRationale
Subscriptions keyed by subscriptionId, not client identityA client could have multiple subscription streams (e.g., different type sets). The spec tags each notification with subscriptionId.
Fan-out iterates all subscriptions per eventExpected subscription count is small (one per connected client that opted in). O(n) is fine. If this ever matters, an inverted index from type to subscriptions would be trivial.
Backpressure is per-streamA slow consumer slows its own SSE stream (TCP backpressure), not other subscriptions. If the write buffer fills, the writer can drop or buffer — this is a per-transport policy.
Subscription management lives on the base McpVersionRouterAll transports need fan-out and cleanup. The transport-specific part is only the SubscriptionWriter implementation (SSE vs. stdout vs. WebSocket frames).
Cleanup on disconnect is automatic via AbortSignalNo explicit "unsubscribe" needed. When the connection drops, the subscription is removed from the map.

8. Custom Transport Author Experience

A developer building a WebSocket or gRPC transport implements two things that form a coupled pair: the Transport (wire protocol) and optionally a McpVersionRouter subclass (classification logic). They choose which eras to support.

8.1 What to implement

8.2 Example: WebSocket transport (dual support)

// Step 1: Implement the wire protocol
class WebSocketServerTransport implements Transport {
    private connections = new Map<string, WebSocket>();

    async start() { /* listen for new WS connections */ }

    async send(message: JSONRPCMessage, options?: TransportSendOptions) {
        const conn = this.getConnection(options?.relatedRequestId);
        conn.send(JSON.stringify(message));
    }

    async close() { /* close all connections */ }

    onmessage?: (message: JSONRPCMessage, extra?: MessageExtraInfo) => void;
    onclose?: () => void;
    onerror?: (error: Error) => void;
}

// Step 2: Implement transport-specific routing logic
class WebSocketVersionRouter extends McpVersionRouter {
    // Track which era each WS connection is using
    private connectionEras = new Map<string, McpEra>();

    classify(message: JSONRPCMessage, meta?: TransportMeta): McpEra {
        const connId = meta?.connectionId;

        // If this connection already picked an era, stick with it
        if (connId && this.connectionEras.has(connId)) {
            return this.connectionEras.get(connId)!;
        }

        // First message on this connection determines the era
        if (isJSONRPCRequest(message)) {
            if (message.method === 'initialize') {
                connId && this.connectionEras.set(connId, 'legacy');
                return 'legacy';
            }
            if (message.params?._meta?.clientCapabilities) {
                connId && this.connectionEras.set(connId, 'modern');
                return 'modern';
            }
        }

        // Default: check body-level signals (shared logic in base class)
        return super.classifyByBody(message);
    }
}

// Step 3: Wire it up
const server = new McpServer({ name: 'my-server', version: '1.0.0' });
server.registerTool('hello', { ... }, handler);

const router = new WebSocketVersionRouter(server, { legacySupport: true });
const transport = new WebSocketServerTransport(wsServer);
await router.serve(transport);

8.3 Example: modern-only gRPC transport (no legacy)

// Simpler: no router subclass needed, just use a trivial classify
class GrpcVersionRouter extends McpVersionRouter {
    classify(): McpEra { return 'modern'; }  // always modern
}

// Or even simpler: McpVersionRouter could provide a factory
const router = McpVersionRouter.modernOnly(server);
const transport = new GrpcServerTransport(grpcServer);
await router.serve(transport);

8.4 Why the transport and router are coupled

The routing signals come from the transport layer. HTTP has headers (Mcp-Method, Mcp-Session-Id), stdio has connection state (first message locks the era), gRPC has metadata. The router needs to inspect these transport-specific signals, which is why:

  • The router classify() method receives TransportMeta alongside the message
  • The transport populates TransportMeta with whatever is relevant (headers, connection ID, etc.)
  • For dual-support transports, the transport itself may need era-aware behavior (e.g., legacy HTTP needs Mcp-Session-Id header echoing, standalone GET SSE)

11. What Changes for Userland Code

The examples below are based on the real SDK examples in examples/server/src/. Tool/resource/prompt registration is identical in before and after — only the transport wiring changes.

11.0 Design principle: building blocks, not just convenience

Problem: If the router absorbs all session management, users who have custom session stores, per-session auth, custom cleanup, or per-session server configs lose that control. We can't predict what custom code they had in mcpPostHandler. Asking them to delete it and trust our implementation is risky.

Solution: The router exposes a two-tier API: a high-level convenience for simple cases, and building blocks that users compose into their existing handler pattern. 
// The HttpVersionRouter API surface
class HttpVersionRouter extends McpVersionRouter {

    // === Tier 1: Building blocks (power users keep their session code) ===

    // Classify an HTTP request by era. Uses headers + body.
    classify(req: Request): McpEra;

    // Handle a modern (2026-06) request. Stateless, no session.
    // Builds context from _meta, dispatches to McpServer, writes response.
    handleModernRequest(req: Request, res: Response): Promise<void>;

    // Create a new legacy session. Returns an object that wraps a frozen
    // Server/Protocol + BridgeTransport. Handlers read from McpServer.
    // Users store/lookup/cleanup these objects however they want.
    createLegacySession(options?: {
        sessionId?: string;           // custom ID, or auto-generated
        eventStore?: EventStore;      // for resumability
    }): LegacySession;


    // === Tier 2: Convenience (absorbs everything for simple cases) ===

    // Handles POST/GET/DELETE, manages sessions internally.
    // Good for getting started. Power users use Tier 1 instead.
    handleRequest(req: Request, res: Response): Promise<void>;

    close(): Promise<void>;
}

// LegacySession: what users store in their session map
// (replaces NodeStreamableHTTPServerTransport in their code)
interface LegacySession {
    readonly id: string;

    // Handle an HTTP request within this session.
    // Same shape as today's transport.handleRequest().
    handleRequest(req: Request, res: Response, body?: unknown): Promise<void>;

    close(): Promise<void>;
    onclose?: () => void;
}

11.1 Stateful HTTP: incremental migration with building blocks

Based on simpleStreamableHttp.ts. The user's session management pattern stays intact — they swap NodeStreamableHTTPServerTransport for LegacySession and add one branch for modern requests.

    const sessionId = req.headers['mcp-session-id'] as string | undefined;
    let transport: NodeStreamableHTTPServerTransport;

    if (sessionId && transports[sessionId]) {
        transport = transports[sessionId];
    } else if (!sessionId && isInitializeRequest(req.body)) {
        transport = new NodeStreamableHTTPServerTransport({
            sessionIdGenerator: () => randomUUID(),
            eventStore: new InMemoryEventStore(),
            onsessioninitialized: sid => { transports[sid] = transport; }
        });
        transport.onclose = () => {
            const sid = transport.sessionId;
            if (sid) delete transports[sid];
        };
        const server = getServer();
        await server.connect(transport);
        await transport.handleRequest(req, res, req.body);
        return;
    } else if (sessionId) {
        res.status(404).json({/*...*/}); return;
    } else {
        res.status(400).json({/*...*/}); return;
    }
    await transport.handleRequest(req, res, req.body);
});

11.2 Stateful HTTP: simple case with convenience API

For users who DON'T have custom session logic, the convenience API absorbs everything:

    legacySupport: true,
    eventStore: new InMemoryEventStore(),
});

    await router.handleRequest(req, res);
});

Internally, handleRequest() is implemented in terms of the same building blocks: classify(), handleModernRequest(), createLegacySession() + an internal session map. Users who need more control use the building blocks directly.

11.6 Codemod feasibility

The migration has two separable phases. Phase 1 is fully codemod-able because LegacySession is API-compatible with the current transport: same handleRequest(req, res, body?), same onclose, same sessionId (as .id). The user's handler pattern stays structurally identical.

Phase 2: Add modern support (manual, human-reviewed)

This is adding new behavior, not migrating old behavior. The user adds one branch at the top of their handler:

    // ... rest of their handler, unchanged from Phase 1 ...

Why Phase 2 is manual: Where to insert the branch depends on the handler structure (middleware? try/catch? auth checks first?). This is new behavior that should be reviewed by a human. But it's one if block — not a rewrite.

12. Open Questions

#QuestionStatus
1Should McpVersionRouter be a separate class or folded into McpServer?Separate. McpServer is the handler registry; router is the serving machinery. Separation keeps McpServer testable without transport.
2Where does HTTP response formatting live (SSE vs JSON)?In HttpVersionRouter (not the base router). The handleRequest() method on HttpVersionRouter owns the HTTP-specific response formatting. The base class works with JSON-RPC messages only.
3How does the modern path handle request/response correlation for stdio?A lightweight correlator inside StdioVersionRouter — simpler than Protocol's full machinery since modern mode has no unsolicited server-to-client requests. Just match outgoing response IDs to incoming request IDs.
4Should we keep McpServer.connect(transport) as a deprecated convenience?Yes, for backwards compatibility. Internally it creates the matching router with legacySupport: true and serves. Deprecation warning guides users to the new API.
5How do handler mutations propagate to the legacy bridge?Resolved (§2.4). Shared HandlerRegistry. McpServer creates the registry, shares it with all Server instances via constructor option. Both modern dispatch and legacy Protocol._onrequest read from the same registry. Handler mutations are instantly visible — no event subscription, no copying, no stale state.
6What about the Client side?Similar pattern: a ClientVersionRouter that probes the server version and then uses either a modern client (per-request _meta, MRTR) or the frozen Client (initialize handshake). Simpler because it picks one era per connection.
7Should the base class provide a classifyByBody() helper for shared body-level checks?Likely unnecessary. HTTP classification uses the Mcp-Method header (present = modern, absent = legacy) — unambiguous by spec design (SEP-2243). Stdio uses first message method (server/discover = modern, initialize = legacy). Body-level fallback may still be useful as a defensive check, but the primary signals are transport-level and clean.
8Should there be a McpVersionRouter.modernOnly(server) factory?Useful for simple cases where custom transports only support modern. Returns a concrete router with classify() => 'modern'. Avoids needing a subclass for the trivial case.
